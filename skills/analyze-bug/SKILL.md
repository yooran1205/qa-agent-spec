# Analyze Bug — 分析缺陷 / 定位根因

## Why — 为什么需要
Bug 分析里最贵的错误不是慢，是**误判根因**：把最显眼或最早刷屏的报错当成起因、把回声当成源头、或者一句"复现不了"就关单。误判的代价是修了症状、放过真因，下次换个形态又爆。

这个 Skill 把根因定位从"看日志找感觉"，换成**证据驱动的确定性方法**——先分离观察与臆断、按时间重建、隔离变量、区分因果与回声，再给出**带置信度和证伪测试**的结论。目的是让同样一份证据，无论谁分析，都能收敛到同一条证据链（结论的置信度可以不同，但推理过程可复查）。

## When — 何时使用
**用它当：**
- 有一份 bug 报告 / 日志 / 复现路径，需要判定根因、严重度、是否真缺陷
- 需要在关单前给出可复查、可证伪的判断，而不是拍脑袋

**不要用它当：**
- 手上完全没有证据（无复现、无日志、无"期望 vs 实际"）→ 先收集证据 / 建复现，别硬分析
- 其实是需求变更包装成的 bug → 那是需求讨论，不是缺陷分析
- 纯性能剖析 / 容量问题 → 那依赖 profiling，另属它类

## Input — 输入契约
**必需输入：** 一份缺陷信息，需能从中提取：
- **观察到什么 vs 期望什么**（现象，去掉报告人的猜测）
- 复现步骤、环境（版本 / 配置 / 数据 / 设备）
- 证据：日志 / 堆栈 / 截图 / 出现频率

**前置条件：** 上述至少具备一项可支撑分析（可复现步骤、或日志、或明确的"期望 vs 实际"）。

**异常输入的处理（结构化退回，不硬猜）：**

| 输入缺陷 | 处理 |
|---|---|
| 只有"出错了"，没说期望是什么 | 停。要"期望行为"，否则无法判断是否缺陷 |
| "复现不了"且无环境/数据信息 | 退回，索要环境 / 账号态 / 数据前置，别直接判"非 bug" |
| 单次出现、无日志无步骤 | 标 `低置信`，索要更多复现证据；不在此基础上下根因结论 |
| 报告里已带作者的根因猜测 | 剥离猜测，只保留原始观察后再分析（猜测常是误判源头） |

## Process — 执行流程
一条证据驱动的流水线。每步的结论都要能回指证据。

1. **分离观察与臆断**：把"观察到什么 / 期望什么"的原始事实，与报告人的解释拆开。只对事实做分析。
2. **定复现类别**：`确定复现 | 偶发(flaky) | 单次`。**未分类前不下根因结论**——偶发/单次要先排时序、环境、数据竞争，别直接当产品 bug。
3. **按时间重建**：把事件按时间排序，找**最早的异常点**，不是最显眼/最靠前刷屏的那条 ERROR。第一条 ERROR 往往是后果。
4. **隔离变量**：什么变了？（近期发布、配置、数据、环境）。能二分就二分，缩小到最小可疑改动。
5. **区分因果与回声**：重复报错先去重，看**独立错误种类**；沿"第一处偏离期望"往回追，而不是往报错最多的地方追。
6. **立假设 + 证伪测试**：把根因写成一句**可检验的断言**，并明确"什么现象会推翻它"（Critic 心态，主动找反例）。
7. **定严重度**：按 **影响面 × 频率** 判，不按日志看着有多吓人判。
8. **成表输出**：根因(带置信度) / 证据链 / 证伪测试 / 严重度 / 修复方向。

> 关键判断点：
> - "最早"≠"最显眼"。醒目的报错通常是后果链末端。
> - "没报错"≠"行为正确"，还要比对期望结果。
> - 让报错消失的改动 ≠ 修好了根因——可能只是把症状盖住了（masking）。

## Output — 完成定义

**输出契约（固定 Schema，列序不变）：**

| 字段 | 内容 |
|---|---|
| 现象 | 观察到什么 vs 期望什么（原始事实） |
| 复现类别 | `确定复现 / 偶发 / 单次` + 置信度 |
| 证据链 | 时间线、最早异常点、发生了什么变化（每条回指证据来源） |
| 根因假设 | 一句可检验断言 + 置信度 `高/中/低` |
| 证伪测试 | 什么现象会推翻这个假设 |
| 严重度 | 影响面 × 频率 得出，非按日志观感 |
| 修复方向 | 针对根因，而非针对症状 |

**完成标准（逐条可核）：**
- [ ] 复现类别已判定，偶发/单次已排除 flaky/时序/环境因素或明确标注未排除
- [ ] 证据链指向**最早异常点**，且每条结论能回指证据
- [ ] 根因是可检验断言，带置信度，并配一条证伪测试
- [ ] 严重度由影响面×频率得出
- [ ] 修复方向针对根因，不是"让报错消失"

## Failure Patterns — 失败模式
| 现象 | 常见误判 | 正确判断路径 |
|---|---|---|
| 日志里第一条/最醒目的 ERROR | 直接判为根因 | 按时间回溯，找**最早**的异常点；醒目报错常是后果 |
| 同类报错刷屏 | 当成很严重、优先处理 | 先去重看**独立错误种类数**，重复是同一根因的回声 |
| 复现步骤走完没报错 | 判"复现不了 / 非 bug" | 查环境/数据前置是否对齐；无错≠行为正确，还要比期望 |
| 偶发失败 | 当产品 bug 上报 | 先判 **flaky**（时序/等待/并发）；稳定复现前不下功能结论 |
| "我这能跑通" | 判对方环境问题、关单 | 隔离环境变量（版本/配置/数据），锁定差异再下结论 |
| 改一下报错就没了 | 判为已修复 | 报错消失≠根因解决，可能是 masking；用证伪测试确认真因 |
| 日志看着很吓人 | 判为高严重度 | 严重度=影响面×频率，吓人的堆栈可能影响面很小 |

## Self-Evaluation
对照 [`spec/evaluation.md`](../../spec/evaluation.md) 自评 **10/10 → best-practice**。

> 诚实说明"可复现"这一维:诊断天生比生成更吃判断，不同分析者对**最终根因的置信度**可能不同。本 Skill 的做法是让**过程**可复现——同一份证据，证据链和"最早异常点"应收敛一致——并把不确定性显式化成`置信度 + 证伪测试`，而不是假装结论唯一。这正是它能拿满分而非回避该维度的方式。

| 维度 | 分 | 依据 |
|---|---|---|
| 完整性 | 2/2 | 五问齐全、共用同一套输出契约与词汇、相互自洽 |
| 触发准确 | 2/2 | 写清"何时用"与三类"何时不用"（含性能剖析、需求变更的边界） |
| 输入契约 | 2/2 | 定义输入 + 前置条件 + 四类异常输入的结构化退回 |
| 失败模式 | 2/2 | 七条，现象/误判/正确路径三列完整——本 Skill 的核心价值所在 |
| 可复现 | 2/2 | 固定 Schema + 复现类别/置信度枚举 + 证据回指 + 证伪测试，让过程可复查 |
