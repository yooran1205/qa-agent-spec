# Generate Test Case — 生成测试用例

## Why — 为什么需要
手写用例最常见的问题不是写得慢，而是**写漏**：正向路径都覆盖了，边界值、异常路径、状态组合却凭记忆挑，挂一漏万。这个 Skill 把"该想到哪些用例"从个人经验变成可复用的覆盖策略，让漏测有据可查，而不是靠人临场记性。

## When — 何时使用
**用它当：**
- 有一份明确的需求 / 功能描述，需要转成可执行的用例集
- 需要在开发前就把验收标准显性化

**不要用它当：**
- 需求本身还不清晰 —— 先去澄清需求（这是另一件事，别用生成用例来掩盖需求缺口）
- 只是要跑已有用例 —— 那是执行，不是生成

## Input — 输入契约
- 一份需求描述：功能点、输入域、状态、期望行为
- **前置条件：** 需求里的验收标准可判定（能说清"什么算对"）。做不到就先回到需求澄清。

## Process — 执行流程
1. **拆输入域**：列出每个输入的取值范围、类型、边界。
2. **正向路径**：每条主流程至少一条通过用例。
3. **边界值**：对每个数值/长度/时间输入，覆盖 min、min−1、max、max+1、空。
4. **异常路径**：非法输入、缺失输入、错误顺序、并发/中断。
5. **状态组合**：涉及状态机的，覆盖关键状态迁移，而非只测单状态。
6. **给每条用例写明期望结果**——没有期望结果的用例是无效用例。

> 关键判断点：
> - 组合会爆炸时，用**等价类 + 边界值**收敛，别穷举；在文档里写清"哪些组合被合并、为什么"，不要静默丢弃。
> - "期望结果"必须来自需求，不能来自当前实现的行为——否则是在给 bug 背书。

## Output — 完成定义
- **交付物：** 一张用例表，每行含：前置条件 / 步骤 / 输入 / **期望结果** / 覆盖类型（正向/边界/异常/状态）。
- **完成标准：** 每个输入域的边界都有对应用例；每条用例都有可判定的期望结果；被合并的组合有说明。

## Failure Patterns — 失败模式
| 现象 | 常见误判 | 正确判断路径 |
|---|---|---|
| 正向用例都通过 | 判定功能没问题 | 正向通过只证明 happy path。缺边界/异常覆盖时，"没问题"是错觉 |
| 期望结果照着当前实现写 | 用例"全过"很安心 | 期望结果的来源必须是需求。跟实现对齐 = 把 bug 固化成基线 |
| 用例数量很多 | 当成覆盖率高 | 数量 ≠ 覆盖。看的是输入域和状态迁移是否被覆盖，不是行数 |
