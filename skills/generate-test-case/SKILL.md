# Generate Test Case — 生成测试用例

## Why — 为什么需要
手写用例最贵的问题不是慢，而是**漏**：正向路径都覆盖了，边界、异常、条件组合、状态迁移却凭记性挑，挂一漏万；而且换个人写、换个模型生成，覆盖面就变一个样，没法复查、没法沉淀。

这个 Skill 把"该想到哪些用例"从个人记性，换成**一套确定性的覆盖技术**——等价类划分、边界值分析、判定表、状态迁移。同样的需求，无论谁执行、用什么模型，都应收敛到基本一致的用例集。漏测从此有据可查，而不是靠临场发挥。

## When — 何时使用
**用它当：**
- 有一份可判定验收标准的需求 / 功能描述，要转成可执行用例集
- 想在开发前把验收标准显性化，作为"什么算对"的契约

**不要用它当：**
- 需求还不清晰、或没有可判定的验收标准 → 先澄清需求（本 Skill 会在 Input 阶段拦截并退回，而不是硬生成）
- 只是要执行已有用例 → 那是执行，不是生成
- 探索性 / 场景化测试 → 那依赖临场判断，不是本 Skill 的结构化覆盖能覆盖的

## Input — 输入契约
**必需输入：** 一份需求描述，需能从中提取：
- 输入项及其取值范围 / 类型 / 长度 / 边界
- 触发条件与它们的组合关系
- 若功能有状态：状态集合与合法迁移
- **可判定的验收标准**（能说清每种情况下"什么算对"）

**前置条件：** 上述四项可提取。**做不到时不生成用例**。

**异常输入的处理（不是猜，而是结构化退回）：**

| 输入缺陷 | 处理 |
|---|---|
| 缺验收标准（说了"做什么"没说"什么算对"） | 停。列出"需明确期望结果"的点，退回澄清 |
| 需求自相矛盾 | 停。列出冲突的两条，退回澄清，不选一条硬做 |
| 边界 / 范围未给（如"密码要够长"没说位数） | 停。列出"待定边界"清单，退回补充 |
| 需求含糊但可判定主干 | 生成主干用例，把含糊点单独标为 `⚠️假设`，注明所依据的假设 |

> 退回时给的是**结构化的缺口清单**，不是一句"需求不清"。缺口清单本身就是交付物的一部分。

## Process — 执行流程
一条命名技术流水线。每步产出的用例都直接进最终输出表。

1. **解析**：从需求抽出「输入项 / 条件 / 状态 / 验收标准」四元组。任一缺失 → 走 Input 的异常处理，不往下走。
2. **等价类划分（EP）**：每个输入项切成有效类 + 无效类，每类取一个代表值。收敛穷举的第一道闸。
3. **边界值分析（BVA）**：对每个有序/有界输入，取 `min−1, min, min+1, max−1, max, max+1`，外加 `空 / null / 类型非法`。边界成对出现，不只测"正确值"。
4. **判定表（Decision Table）**：多个条件组合影响结果时，列条件组合 → 期望动作。组合爆炸时用 **pairwise（成对覆盖）** 收敛，并在表末**写明哪些组合被合并、依据什么**——不静默丢弃。
5. **状态迁移（State Transition）**：有状态的功能，覆盖每条合法迁移，外加关键非法迁移（如"锁定态下即便操作合法也应拒绝"）。
6. **绑定期望结果**：逐条用例的期望结果**只能来自验收标准**，并在「需求依据」列回指是哪一条。禁止照当前实现行为回填。
7. **按输出契约成表**，套决定性规则排序与编号。

> 关键判断点：
> - 组合爆炸 → EP + BVA + pairwise 三件套收敛，**别穷举**；合并要留痕。
> - 期望结果来源 = 需求。跟实现对齐 = 把 bug 固化成基线。
> - "没报错"≠"行为正确"，期望结果必须是可判定的具体值，不是"正常"。

## Output — 完成定义

**输出契约（固定 Schema，列与列序不变）：**

| ID | 覆盖类型 | 前置条件 | 操作步骤 | 输入 | 期望结果 | 需求依据 |
|---|---|---|---|---|---|---|

- `覆盖类型` 枚举：`正向 | 边界 | 异常 | 组合 | 状态`
- `需求依据` 回指验收标准的条目（如 `AC-3`），做到用例可追溯到需求。

**决定性规则（保证可复现的关键）：**
- **ID**：`TC-<模块缩写>-<三位序号>`，序号从 001 连续递增。
- **排序**：先按覆盖类型 `正向→边界→异常→组合→状态`，同类型内按输入项在需求中出现的顺序。
- **合并留痕**：pairwise / 等价类合并掉的组合，在表末「合并说明」里逐条列出，可回溯。

**完成标准（逐条可核）：**
- [ ] 每个输入项的每条边界都有对应用例（BVA 成对齐全）
- [ ] 每个多条件功能有判定表，组合要么覆盖、要么在「合并说明」里有交代
- [ ] 每条状态迁移有用例，含至少一条关键非法迁移
- [ ] 每条用例的期望结果可判定，且「需求依据」非空
- [ ] 输出严格遵循 Schema、枚举、ID、排序规则

## Failure Patterns — 失败模式
| 现象 | 常见误判 | 正确判断路径 |
|---|---|---|
| 正向用例全过 | 判定功能没问题 | 正向通过只证明 happy path；缺边界/异常/状态覆盖时，"没问题"是错觉 |
| 期望结果照当前实现写 | 用例"全过"很安心 | 期望的唯一来源是验收标准。跟实现对齐 = 把 bug 固化成基线（`需求依据`列就是防这个的） |
| 用例数量很多 | 当成覆盖率高 | 数量 ≠ 覆盖。看的是输入域边界、条件组合、状态迁移是否被覆盖，不是行数 |
| 条件组合太多就只挑几条测 | "主要场景测了就行" | 静默挑选=静默漏测。用 pairwise 收敛并在「合并说明」留痕，让漏的部分是显式决策 |
| 需求含糊就按自己理解生成 | 先做出来再说 | 含糊点会变成错误基线。可判定主干照做，含糊处标 `⚠️假设` 或退回澄清，别替需求做主 |
