# Review PRD — 评审需求文档

## Why — 为什么需要
需求缺陷在写文档时最便宜、上线后最贵。QA 评审 PRD 的价值，是在写一行代码前就揪出"不可测 / 有歧义 / 漏边界 / 没验收标准"的问题。

手动评审最容易漏的，是**没写出来的那部分**——异常流被沉默省略、边界没提、验收标准含糊。人通读时会不自觉地"脑补"作者的意图，从而看不见缺口。这个 Skill 把 PRD 评审从"通读挑刺"换成**固定维度的结构化扫描**，专门盯隐性缺失，让"哪里没写清"有据可查、有优先级、有 gate 结论。

## When — 何时使用
**用它当：**
- 有一份成文的需求 / PRD，需要在开发前从质量与可测性角度评审
- 需要给出"能否进入开发"的门槛判断（gate），而不只是主观感受

**不要用它当：**
- 需求还停在一句话 / 口头阶段 → 先补成可评审的文档
- 评审的是技术方案 / 架构设计 → 那是架构评审，另属它类
- 评审的是代码 → 用代码评审，不是本 Skill

## Input — 输入契约
**必需输入：** 一份成文需求，需含可评审的功能主干（功能描述 / 用户故事 / 验收标准 / 边界，哪怕不全）。

**前置条件：** 文档可读、有可评审粒度的内容。

**异常输入的处理：**

| 输入缺陷 | 处理 |
|---|---|
| 只有标题 / 一句话，无可评审主干 | 停。退回，要求补到可评审粒度（否则评审=替作者写需求） |
| 有主干但缺验收标准 / 边界 | **不退回**——这正是评审要发现的问题，作为发现项列出（缺失即产出） |
| 存在多个矛盾版本 | 停。要求指定评审哪一版，不擅自选 |

> 注意 Input 的判断线：**文档"薄"到没法评审 → 退回；文档有主干但有缺口 → 照评，缺口就是发现。** 别把"发现缺口"和"无法评审"搞混。

## Process — 执行流程
按**固定七维**逐一扫描，每维找缺口。同一份 PRD、不同评审者扫同一组维度，发现集应高度重叠。

1. **可测性**：每条需求能否判定"什么算对"？有无可观测、可判定的验收标准？
2. **完整性**：正常流程之外，异常流 / 边界 / 空态 / 并发 / 权限态覆盖了吗？
3. **明确性**：有没有"友好地""合理地""尽快""常见格式"这类**不可判定的模糊词**？
4. **一致性**：文档内部是否自相矛盾？与现有功能 / 已有约定是否冲突？
5. **边界与数据**：输入域、取值范围、上下限、非法输入的行为定义了吗？
6. **风险与影响面**：这个需求动了什么？对现有功能 / 数据 / 兼容性的影响评估了吗？能回滚吗？
7. **依赖与前置**：依赖的上游 / 外部系统 / 开关 / 数据前置说清楚了吗？

> 关键判断点：
> - 重点找**没写出来的**（隐性缺失），而不是只评"写出来的对不对"。
> - 不可判定的形容词（友好/合理/快）= 不可测，必须要求量化。
> - 无发现的维度也要显式标"已检查、无问题"——**"没提" ≠ "已查过没问题"**。

## Output — 完成定义

**输出契约（固定 Schema）：** 一张发现表 +一条 gate 结论。

| ID | 维度 | 位置 | 问题 | 风险（为什么是问题） | 建议（怎么补） | 严重度 |
|---|---|---|---|---|---|---|

- `维度` 取自上面七维之一；`位置` 回指 PRD 的章节 / 段落。
- `严重度` 枚举：`阻塞 | 重要 | 建议`
  - **阻塞**：不解决无法开始开发 / 测试（如整块无验收标准）
  - **重要**：会导致漏测或返工，需在开发前澄清
  - **建议**：优化项，不阻塞

**gate 结论**：`通过 / 有条件通过 / 打回`——只要存在 `阻塞` 项即为"打回"。

**决定性规则（保证可复现）：**
- **ID**：`R-<维度缩写>-<序号>`（如 `R-测-01`）。
- **排序**：先按七维顺序，同维度内按 PRD 位置先后。
- **全维度留痕**：七维每一维都要在输出里出现，无发现的标 `✓ 已检查、无问题`，杜绝"没列=没查"的歧义。

**完成标准（逐条可核）：**
- [ ] 七维全部扫过并在输出留痕（含标注无问题的）
- [ ] 每个发现都有 位置 / 风险 / 建议 / 严重度
- [ ] 给出 gate 结论，且存在阻塞项时结论为"打回"
- [ ] 输出严格遵循 Schema、严重度枚举、ID 与排序规则

## Failure Patterns — 失败模式
| 现象 | 常见误判 | 正确判断路径 |
|---|---|---|
| PRD 写得又长又详细 | 判为"需求很清晰" | 长 ≠ 完整。按七维查缺口，重点看**没写的**部分 |
| 只评"写出来的对不对" | 漏掉隐性缺失 | 主动查异常流 / 边界 / 空态是否被沉默省略 |
| "友好地 / 合理地 / 尽快 / 常见格式" | 当成正常描述放过 | 不可判定的词=不可测，必须要求量化或枚举 |
| 正常流程写得很清楚 | 判为"可以开测" | happy path 清楚 ≠ 验收标准可判定；缺异常流验收=未来漏测 |
| 扫出一堆小问题 | 全标高优先级 | 用 阻塞/重要/建议 分级，别让噪声淹没真正的 gate 问题 |
| 只盯着本文档看 | 判为"内部自洽即可" | 一致性要**跨现状**看：与已有功能 / 约定是否冲突 |

## Self-Evaluation
对照 [`spec/evaluation.md`](../../spec/evaluation.md) 自评 **10/10 → best-practice**。

> 关于"可复现"：评审是评估型，属**过程可复现**为主——固定七维保证不同评审者扫的是同一组维度，发现集应高度重叠。严重度分级和 gate 规则把"能否开发"这个结论也钉成可判定的（存在阻塞即打回），因此结论层也稳定。

| 维度 | 分 | 依据 |
|---|---|---|
| 完整性 | 2/2 | 五问齐全、共用同一套输出契约与词汇、相互自洽 |
| 触发准确 | 2/2 | 写清"何时用"与三类"何时不用"（含架构评审、代码评审的边界） |
| 输入契约 | 2/2 | 定义输入 + 前置条件 + 异常输入处理，并厘清"退回 vs 发现"判断线 |
| 失败模式 | 2/2 | 六条，现象/误判/正确路径三列完整 |
| 可复现 | 2/2 | 固定七维 + 发现表 Schema + 严重度枚举 + 全维度留痕 + gate 规则，过程与结论均可复查 |
