# Critic Pattern — 评判者模式

> 一个独立的 Agent，唯一职责是**反驳**主 Agent 的结论。

## Why — 解决什么协作问题
一个 Agent 得出结论后再自查，天然偏向**证实自己**（motivated reasoning）——它已经"信"了这个结论。而 QA 里最贵的错误恰恰是**自信的错误结论**：假"通过"、误判根因、把真 bug 判成"非 bug"。

Critic Pattern 引入一个**角色分离**的 Agent：它不产出结论，只负责找结论错在哪。因为它对"结论正确"不投入、不留恋，能抓到自查抓不到的错——这不是模型能力问题，是**立场问题**。

## When — 何时使用
**用它当：**
- 错误结论代价高：根因判定、发布 gate、"这是不是真缺陷"
- 结论"看着对但可能错"——最危险的那类
- 你能说出一条**证伪测试**（能说清"什么会推翻它"，就适合上 Critic）

**不要用它当（避免过度设计）：**
- 低风险 / 易回滚的产出——多一个 Agent 不值
- 纯机械、有确定答案的任务——直接校验答案，别组织辩论
- 没有独立证据可供 Critic 推理——garbage in，Critic 也只是走过场

## Structure — 结构

```
                    ┌─────────────┐
   同一份原始证据 ──▶│  Producer   │──▶ 结论 + 证据 + 证伪测试
        │           └─────────────┘
        │                                  │
        └──────────────┐                   ▼
                       ▼            ┌─────────────┐
                （独立拿到原始证据）│   Critic    │  职责：反驳
                                   └─────────────┘
                                          │
                              ┌───────────┴───────────┐
                              ▼                       ▼
                        证据驳不倒 → 结论存活      落地反例 → 打回 Producer
```

关键：Critic **拿到的是同一份原始证据，不是 Producer 的解释**；它被要求主动反驳，**存疑即判"驳回"**，从独立证据推理，而非顺着 Producer 的思路走。

## Mechanics — 怎么跑
1. **Producer** 产出：结论 + 证据 + （最好）一条证伪测试。
2. **Critic** 拿到**同一份原始证据**（不是 Producer 的推理链），独立重推。
3. Critic 的指令：找"这个结论错了"的最强理由。**存疑时默认判驳回**，而不是默认放过。
4. **裁决**：Critic 拿不出证据反驳 → 结论存活；Critic 给出具体反例 → 打回 Producer。
5. **升级（高风险场景）**：N 个 Critic，各用不同视角（正确性 / 能否复现 / 有无其他根因），多数判驳回则否决——视角**多样**比数量多更重要，冗余的同质 Critic 抓不到互补的失败模式。

## Failure Patterns — 这个模式自己的坑
| 现象 | 常见误判 | 正确做法 |
|---|---|---|
| Critic 看了 Producer 的推理链 | 被带着走，盖章放行 | 只给 Critic **原始证据**，不给结论的论证 |
| Critic 默认"看起来没问题" | 又变成一次证实 | 指令上**默认判驳回**，奖励它找漏洞 |
| 把单个 Critic 当真理 | 单点失效 | 高风险场景用 N 个多样视角 Critic |
| Critic 和 Producer 同模型同 prompt | 盲点相关，一起漏 | 换视角 / 换立场——**独立**才是这个模式的意义 |
| Producer↔Critic 无限拉扯 | 卡住不收敛 | 限定回合数；K 轮未决 → 升级给人，附双方立场 |

## Relationship — 和 Skill / 其他模式的关系
- **和 Skill 层**：`analyze-bug` 里的"证伪测试"，就是把 Critic **内建进单个 Skill** 的迷你形态。Critic Pattern 是同一思想放大到**多 Agent 编排**。
- **和 Review Loop**：常组合——Producer 产出 → Critic 反驳 → 修正，循环到收敛。Critic 提供"该不该再改一轮"的判据。

## Example — 一个通用样例
沿用 `analyze-bug` 的"偶发下单失败"：
- **Producer 结论**：根因 = 上游超时后缓存回退返回 null 价格，未校验即崩在 buildReceipt。置信度中。证伪测试：构造超时+缓存未命中，若 total 非 null 则推翻。
- **Critic**（只拿到原始日志，不看 Producer 推理）：主动找替代解释——"会不会是 buildReceipt 自身的并发问题，跟价格无关？"→ 跑证伪测试：在价格非 null 时也复现 NPE 吗？
  - 复现不了 → Producer 结论**存活**（Critic 反而强化了它）。
  - 能复现 → 落地反例，**打回** Producer 重查。

Critic 没有"帮忙确认"，它在**认真找反例**——这正是它拦住误判的方式。
