# Failure Patterns — 失败模式

这是本项目最不一样的地方，也是它最难被抄走的地方。

"怎么做"到处都是。**"哪里最容易判断错、以及正确的判断路径是什么"** —— 只有真正干过 QA、踩过坑的人写得出来。把这些从资深测试的脑子里挖出来、写成结构化的知识，是这个规范存在的核心理由。

## 为什么它比"流程"更值钱

一份 Skill 的 Process 描述的是"一切顺利时怎么走"。但测试的价值恰恰在**不顺利**的时候——现象具有误导性、根因藏在噪声后面、看似通过其实是假阳性。

一个只会走 Process 的 Agent，会自信地给出错误结论。一个内化了 Failure Patterns 的 Agent，会在该怀疑的地方怀疑。

## 结构

每条失败模式用三列表达：

| 现象（看到什么） | 常见误判（容易当成什么） | 正确判断路径（该怎么判） |
|---|---|---|

三列缺一不可：
- 只写"现象"→ 没帮助。
- 写了"误判"没写"正确路径"→ 告诉 Agent 别踩坑，却没告诉它往哪走。

## 示例（通用、不含任何组织特定信息）

**Skill: 日志分析 / 定位根因**

| 现象 | 常见误判 | 正确判断路径 |
|---|---|---|
| 日志里第一条 ERROR 很醒目 | 直接判定为根因 | 按时间回溯：第一条 ERROR 往往是后果。找**最早的异常时间点**，而不是最显眼的那条 |
| 大量重复报错刷屏 | 当成严重问题优先处理 | 重复常是同一根因的回声。先去重再看**独立错误种类数** |
| 复现步骤走完没报错 | 判定为"无法复现 / 非 bug" | 检查是否环境/数据前置没对齐；无错 ≠ 行为正确，还要比对**期望结果** |

**Skill: UI 自动化断言**

| 现象 | 常见误判 | 正确判断路径 |
|---|---|---|
| 元素存在，断言通过 | 判定功能正常 | 存在 ≠ 可见 ≠ 可交互。分层断言：存在 → 可见 → 命中期望值 |
| 用例偶发失败 | 当成产品 bug 上报 | 先判**flaky**：是否等待/时序问题。稳定复现前不下功能结论 |

> 上面的例子刻意不依赖任何具体工具或项目——它们在任何测试栈下都成立。这既是失败模式该有的抽象层级，也是保证规范可公开、可复用的方式。

## 怎么收集失败模式

最好的来源是**复盘**：每次线上事故、每次误报漏报，问一句"当时最容易怎么判错？正确路径是什么？"，把答案沉淀成一行。

日积月累，这张表就是团队测试经验的工程化形态。
