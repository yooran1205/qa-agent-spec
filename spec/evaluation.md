# Evaluation — 评测体系

一个 Skill 好不好，不能靠感觉。这份评测给出可打分的维度，用来判断成熟度、发现短板、指导迭代。

## 五个维度

每个维度 0–2 分，满分 10。

| 维度 | 0 分 | 1 分 | 2 分 |
|---|---|---|---|
| **完整性** | 缺 Why/When/Input/Process/Output 之一 | 五问齐全但薄弱 | 五问清晰且相互自洽 |
| **触发准确** | 没写 When，或边界模糊 | 写了何时用 | 同时写清"何时用"与"何时不用" |
| **输入契约** | 输入未定义 | 定义了输入 | 定义输入 + 前置条件 + 异常输入处理 |
| **失败模式** | 没有 | 列了坑但没给正确路径 | 现象 / 误判 / 正确路径三列完整 |
| **可复现** | 换人换模型，过程和结论都发散 | 过程大体一致 | 过程稳定收敛，且不确定性被显式化（见下） |

### "可复现"的两层：过程可复现 vs 结论可复现

按维度直接打分，会不公平地压低诊断/判断型 Skill——它们的结论天生带不确定性。所以把可复现拆两层：

- **过程可复现**：同一份输入，不同执行者走出的**推理路径 / 证据链 / 中间产物**应当收敛一致。**这是硬要求，任何 Skill 都要满足。**
- **结论可复现**：最终产出是否逐字一致。**要求随 Skill 类型不同：**

| Skill 类型 | 结论可复现的要求 | 拿 2 分的标准 |
|---|---|---|
| **生成型**（如 generate-test-case） | 结论应高度一致 | 过程收敛 **且** 结论稳定可预期 |
| **判断/诊断型**（如 analyze-bug） | 结论可带概率，不强求唯一 | 过程收敛 **且** 把不确定性**显式化**（置信度 + 证伪测试），而不是假装唯一答案 |

判断型 Skill 若强行输出"唯一确定结论"，反而应扣分——那是在掩盖真实的不确定性。**显式的不确定性 > 虚假的确定性。**

## 分数 → 成熟度

| 总分 | 建议成熟度 |
|---|---|
| 0–3 | prototype |
| 4–5 | stable |
| 6–7 | production |
| 8–9 | recommended |
| 10 | best-practice |

## 怎么用

1. 写完一个 Skill，对着五维自评，把弱项补上。
2. 让另一个人（或另一个模型）不看你的意图、只按 Skill 执行一遍——比对的是**推理路径与证据链是否收敛**（过程可复现），而非只看最终输出是否逐字一致。这是"可复现"维度最真实的检验。
3. 分数写进 PR，成熟度标进 metadata，形成可追溯的演进记录。

> 评测本身也应该框架无关：它衡量的是 Skill 作为"知识单元"的质量，而不是它在某个平台上跑得多顺。
