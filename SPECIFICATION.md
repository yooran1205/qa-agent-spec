# Skill Specification

一个 QA Skill 是一份让 AI Agent 能可靠执行某项测试工作的说明书。它不是一段 prompt，而是一个有结构、可评测、可版本化的单元。

本规范定义一个 Skill 必须包含哪些部分，以及每部分回答什么问题。

## 核心结构：五问

每个 Skill 都要回答这五个问题。缺一个，它就退化成一段 prompt。

| 部分 | 回答的问题 | 为什么重要 |
|---|---|---|
| **Why** | 为什么需要这个 Skill？不做会怎样？ | 没有 Why，Skill 只是被机械执行，无法在边界情况下自己判断取舍 |
| **When** | 什么时候用它、什么时候**不**用？ | 触发边界不清，Agent 会在错误场景滥用它 |
| **Input** | 需要什么输入？前置条件是什么？ | 输入契约不清，是 Agent 失败最常见的源头 |
| **Process** | 怎么一步步执行？关键判断点在哪？ | 这是 Skill 的主体，但**不是**最有价值的部分 |
| **Output** | 怎样算完成？交付物长什么样？ | 没有完成定义，Agent 不知道何时停、交付什么 |

再加两个让它区别于"漂亮文档"的部分：

| 部分 | 回答的问题 |
|---|---|
| **Failure Patterns** | 这件事最容易错在哪？哪些是误判？正确的判断路径是什么？（见 [`spec/failure-patterns.md`](./spec/failure-patterns.md)） |
| **Metadata** | 成熟度、版本、依赖假设（见 [`spec/metadata.md`](./spec/metadata.md)） |

## 一个反例

❌ **这不是一个 Skill：**
> "分析这段日志，找出报错原因。"

它没说什么时候该用、输入要什么格式、什么算分析完、以及——最关键的——**哪些报错看着像根因其实是噪声**。换个模型、换个人，结果完全不同。

✅ **这是一个 Skill：** 见 [`skills/generate-test-case/SKILL.md`](./skills/generate-test-case/SKILL.md)。

## 框架无关（Framework Agnostic）

Skill 描述的是**测试意图和判断逻辑**，不是某个平台的 API 调用。

- ✅ "对跨 App / 系统弹窗 / 权限框的交互，用能操作系统 UI 层的方式驱动"
- ❌ "调用 `xxx-cli connectedAndroidTest --device yyy`"

好处有二：一是任何团队、任何工具链都能落地；二是**这条约束天然过滤掉了组织特定的实现细节**，让规范保持纯粹和可公开。

写 Skill 时的自检：**"如果换一个工具链，这条还成立吗？"** 成立才写进规范；只在某套内部工具下成立的，属于实现层，放到你自己的私有适配层里，不进这个仓库。

## 反静默省略（No Silent Omission）

一个通用约定，对所有做"覆盖"的 Skill 都成立：**凡是逐项检查 / 逐条覆盖的动作，每一项都要在输出里留痕——包括没发现问题的、和被有意跳过的。**

沉默有歧义：一个维度没出现在输出里，读者无法区分它是"查过、没问题"还是"根本没查"。而这两者对质量的意义天差地别。

落地成三条：
- **查过无问题** → 显式标 `✓ 已检查、无问题`，而不是不写。
- **有意跳过 / 合并** → 写明跳了什么、依据什么（如用例的 pairwise 合并、诊断中排除的复现类别）。
- **禁止静默截断** → 因规模只覆盖了 top-N、抽样、或未穷举时，明确说清"覆盖了多少、放弃了什么"。

三个现役 Skill 都体现了它：`generate-test-case` 的合并留痕、`analyze-bug` 的复现类别排除、`review-prd` 的全维度留痕。它之所以是规范级而非某个 Skill 的细节，是因为**"沉默的省略被当成已覆盖"是所有 QA 工作共通的漏测根源**。

## 一个 Skill 的目录约定

```
skills/<skill-name>/
├── SKILL.md          # 遵循本规范的主体
└── examples/         # 真实输入 → 期望输出的样例
```
